# Backend Спецификация (Golang)

Этот документ написан специально для Backend-разработчика с фокусом на простоту реализации в сжатые сроки хакатона. 

Архитектура построена по принципу **Thin Client**, то есть фронтенд ничего не вычисляет и не агрегирует. Ваша задача на Go — обращаться к базе данных, упаковывать всё в удобный JSON и отдавать фронтенду, чтобы он просто их расставил по странице.

> **Важно:** 
> - Сложная настройка таблиц базы данных в Supabase и сложные алгоритмы Data Analysis/Машинного Обучения **не входят** в ваши задачи. 
> - Нам нужен максимально простой, надежно работающий алгоритм `if/else`, который "под капотом" не упадет во время демонстрации.

## Стек и Рекомендуемые Библиотеки (Go)
Для старта инициализируйте проект: `go mod init neoflex-lms`. Чтобы не изобретать велосипед, берите готовые стандарты индустрии:
- **Роутинг HTTP-сервера:** Фреймворк `github.com/gin-gonic/gin` — он очень прост в освоении для новичка, содержит кучу готовых решений и идеально подходит для написания REST API.
- **Работа с PostgreSQL (Supabase):** Пакет `github.com/jackc/pgx/v5` или ORM `gorm.io/gorm`. С ними легко писать запросы к БД.
- **Docker-песочницы (Sandbox):** API-клиент `github.com/docker/docker/client` (Официальный Docker SDK для запуска контейнеров программно из кода Go).
- **Парсер SQL (Для Checker'a):** Сторонняя библиотека, например `github.com/xwb1989/sqlparser`, чтобы парсить запросы студента и избегать краха базы.
- **Data exchange:** Никаких сложных gRPC/Protobuf. Обычный стандартный `encoding/json`.

## Простые алгоритмы работы (Бизнес-логика)

### 1. Как работает Code/SQL Checker (Самая важная Wow-фича)
Вместо сложного изолированного рантайма для кучи языков, давайте для хакатона сделаем идеальную проверялку SQL-запросов (так как это востребовано у HR):
1. Фронтенд присылает JSON: `{"user_id": "123", "task_id": "45", "user_code": "SELECT * FROM users;"}`.
2. Простая защита (Валидация): Вы проверяете строку студента с помощью `sqlparser` или просто `strings.Contains(strings.ToUpper(code), "DROP")`. Если в запросе есть слово `DROP`, `DELETE` или `UPDATE` (когда таска подразумевала `SELECT`), то сразу возвращаете ошибку: `{"status": "failed", "console": "Access Denied: Запрещенные команды"}`.
3. Если строка безопасна: Go выполняет `db.Query(user_code)` подRead-Only пользователем БД.
4. Вы сравниваете полученный результат (кол-во строк и столбцов) с эталоном.
5. При успехе возвращаете JSON: `{"status": "success", "console": "Success: 5 rows matched"}`.

### 2. Как работает Аналитика (Прогресс и Матрица)
Мы пропускаем сложный анализ данных для экономии времени. Схема упрощена:
- В БД есть таблица `USER_COMPETENCIES`, в которой уровень владения навыком (SQL, Python) сохранен в виде числа (от 0 до 100).
- При получении `success` от Checker'а, Backend просто выполняет апдейт: `UPDATE user_competencies SET level = level + 5 WHERE user_id = X AND competency_id = Y`. Дальше фронтенд сам запросит новые числа и санимирует радарный график!

## Timeline реализации (Пошаговый план Backend-разработчика)

Шаги разбиты на микро-задачи. Вы можете спокойно идти по ним сверху-вниз.

### Шаг 1: Поднятие сервера и API Gateway (Часы 1-3)
*Задача: вывести приложение в сеть и вернуть первый JSON фронтендеру.*
- [ ] **Микрозадача 1.1:** Установить Gin: `go get -u github.com/gin-gonic/gin`.
- [ ] **Микрозадача 1.2:** Создать файл `main.go`. Инициализировать роутер `r := gin.Default()`.
- [ ] **Микрозадача 1.3:** Настроить CORS-middleware в Gin (очень важно, иначе Фронт на React упрется в браузерную защиту и не сможет делать запросы к Go!). Разрешить заголовки `Content-Type` и `Authorization`.
- [ ] **Микрозадача 1.4:** Добавить эндпоинт-заглушку `GET /ping`, возвращающий `{"status": "ok", "message": "pong"}`. Дать отмашку фронтендеру сделать первый вызов.
- [ ] **Микрозадача 1.5:** Подключить базу: Инициализировать `pgxpool` соединение к Supabase с использованием строки аутентификации из `.env`.

### Шаг 2: Эндпоинты агрегации (Read-Only API) (Часы 3-6)
*Задача: Отдать Фронту массивы данных, чтобы он мог верстать дашборды.*
- [ ] **Микрозадача 2.1 - Каталог:** Эндпоинт `GET /api/courses`. Делает `SELECT * FROM courses WHERE is_published = true`. Отдает список структур в JSON.
- [ ] **Микрозадача 2.2 - Рецепт Курса:** `GET /api/courses/:id/lessons`. Go делает join таблицы Курса и Уроков, сортирует по `order_index` и отдает готовое дерево.
- [ ] **Микрозадача 2.3 - Матрица скиллов:** `GET /api/users/me/competencies`. Берем `user_id` из токена, достаем скиллы из базы, возвращаем массив `{ "name": "SQL", "level": 25 }`. 

### Шаг 3: Checker логика (Часы 6-12)
*Главная фича, за которую жюри ставят баллы.*
- [ ] **Микрозадача 3.1:** Роут `POST /api/check`. Создать Go Struct для входящего payload.
- [ ] **Микрозадача 3.2:** Провести валидацию присланного SQL кода студента (как описано в Алгоритме №1).
- [ ] **Микрозадача 3.3:** Выполнить прогон скрипта. Если база вернула DB Error — перехватить её и аккуратно записать текст ошибки в ответ JSON `console_output`, чтобы фронт показал её пользователю.
- [ ] **Микрозадача 3.4:** При успехе выполнить Insert в БД `SUBMISSIONS`. Запустить небольшую функцию-счетчик (добавить баллы за старания в матрицу пользователя `USER_COMPETENCIES`).

### Шаг 4: Sandbox Docker Manager (Часы 12-16)
*Вторая Wow-фича для презентации.*
- [ ] **Микрозадача 4.1:** Установка SDK `go get github.com/docker/docker/client`.
- [ ] **Микрозадача 4.2:** Создать роут `POST /api/sandbox/start`. В теле берется image name (например, `nginx`).
- [ ] **Микрозадача 4.3:** Используете `cli.ContainerCreate` и `cli.ContainerStart`. Просите докер пробросить на хост случайный рандомный порт.
- [ ] **Микрозадача 4.4:** Достаете этот порт, собираете URL вида `http://vps-server-ip:порт` и отдаете JSON на Фронт.
- [ ] **Микрозадача 4.5:** (Самый простой Garbage Collector): Просто запускаете горутину `go func() { time.Sleep(30*time.Minute); cli.ContainerRemove() }()` — и контейнер автоматически уничтожится через полчаса, чтобы не засорить вашу виртуалку.

### Шаг 5: SVN Листенер и Поддержка Фронта (Финальные часы)
*Точечные улучшения*
- [ ] **Микрозадача 5.1:** Сделать роут `POST /webhooks/git`. Принимает JSON от гитхаба или SVN.
- [ ] **Микрозадача 5.2:** Простой алгоритм: если в JSON хука есть событие "Merge PR", искать в названии коммита TaskID (через Regex) и проставлять галочку `success` данному `user_id` в базе данных.
- [ ] **Микрозадача 5.3:** Пройтись по всем роутам и убедиться, что ни один не приводит к крашу (`panic()`). Ошибки типа 404/400 должны возвращаться в виде адекватного JSON: `{"error": "Пользователь не найден"}` — это сильно упростит жизнь фронт-разработчику.
