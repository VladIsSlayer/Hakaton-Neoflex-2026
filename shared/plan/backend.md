# Backend Спецификация (Golang)

Этот документ написан специально для Backend-разработчика с фокусом на кристальную простоту реализации в сжатые сроки хакатона. 

Архитектура построена по принципу **Thin Client**: фронтенд ничего не вычисляет. Ваша задача на Go — обращаться к базе данных, осуществлять аутентификацию и отдавать данные с учетом ролевой модели (Юзеры и Модераторы), аккуратно упаковывая всё в формат JSON.

> **Важно:** 
> Сложные алгоритмы анализа данных **не входят** в ваши задачи. Нужен максимально простой, железобетонно работающий код, который не подведет во время презентации.

## Стек и Рекомендуемые Библиотеки (Go)
Инициализация проекта: `go mod init neoflex-lms`. 
- **Роутинг и Авторизация:** Фреймворк `github.com/gin-gonic/gin` — идеально для REST API и интуитивно понятного написания Middleware (промежуточного ПО для проверки токенов).
- **Работа с PostgreSQL:** Пакет конфигурации `github.com/jackc/pgx/v5` или базовая ORM `gorm.io/gorm`.
- **Проверка кода (Система Judge0):** Для проверки кода студента мы будем использовать сторонний открытый API — **Judge0**. Эта платформа сама изолирует код и защищает от уязвимостей, поддерживая >50 языков.
- **Работа с JWT токенами:** Библиотека `github.com/golang-jwt/jwt/v5` для проверки логина и идентификации ролей пользователя.

## Подробные алгоритмы работы (Бизнес-логика)

### 1. Как работает Аутентификация и Ролевая Модель
У нас есть две главные бизнес-роли: **Студенты (Users)** и **Модераторы/HR**.
1. **Процесс Логина:** Бэкенд реализует ручку `POST /api/auth/login`. Студент присылает email/пароль. Go ищет пользователя в БД, хэширует/сравнивает пароль. В случае успеха Go генерирует **JWT-токен**, "зашивая" туда payload: `{"user_id": 123, "role": "student"}` и возвращает его фронтенду.
2. Фронтенд сохраняет токен и шлет заголовок `Authorization: Bearer <token>` во все приватные запросы (Например, в Личный кабинет).
3. **Безопасность (Middleware):** Go-бэкенд использует Middleware-функцию в Gin, которая перехватывает все защищенные запросы. Она расшифровывает токен. Если токена нет — отдает ошибку `401 Unauthorized`. Если всё окей — кладет `user_id` и `role` в контекст запроса.
4. **Защита ручек (RBAC):** Если модератор шлет запрос `POST /api/courses/create` (создать новый модуль), бэкенд вытаскивает роль из токена: `if role != "moderator" { return 403 Forbidden }`. Таким образом, студенты не смогут ломать и изменять каталог курсов.

### 2. Как работает Code Checker (Интеграция с Judge0)
Писать свой локальный компилятор вы не будете, используем Judge0 как API.
**Алгоритм:**
1. Фронтенд присылает JSON: `{"task_id": "45", "user_code": "print('hello')", "language_id": 71}`. *`user_id` вы берете из JWT-токена в целях безопасности (чтобы кто-то не отправил решение от чужого имени)!*
2. Go-сервер через стандартный HTTP-клиент отправляет запрос к Judge0 API (эндпоинт `/submissions`).
3. Judge0 безопасно выполняет этот код и отвечает JSON-объектом, содержащим поле `stdout` (чистый текстовый вывод консоли от кода пользователя).
4. Go делает SELECT из БД: берет поле `reference_answer` (эталонный вывод) для задачи "45".
5. Банальное сравнение: `if strings.TrimSpace(stdout) == strings.TrimSpace(reference_answer)`. 
6. Если сошлось — отдаете фронтенду: `{"status": "success", "console": stdout}`.

### 2. Как работает Аналитика (Прогресс и Матрица Компетенций)
**Подробный ход работы бекенда при начислении баллов:**
Никакой сложной высшей математики! Аналитика сводится к счетчикам очков. Когда Чекер в Алгоритме №1 сгенерировал `success` для студента, Go-бэкенд продолжает работу:
1. Вы запрашиваете у БД: к какому навыку или компетенции привязана эта задача? (`SELECT competency_id FROM tasks WHERE id = 45`).
2. Проверяете наличие записи у юзера: `SELECT level FROM user_competencies WHERE user_id = 123 AND competency_id = X`.
3. **Логика начисления:** Если записи еще нет — делаем `INSERT` с начальным `level = 10` (баллов). Если запись уже есть — делаем `UPDATE level = level + 10`. *Ограничение: `if level > 100 { level = 100 }`*.
4. **Общий прогресс курса:** Считаете метрику завершенности. Запрашиваете кол-во _всех_ задач в курсе и кол-во `success` задач у данного юзера в таблице `SUBMISSIONS`. Формула: `(Решено / Всего) * 100%`.
5. Делаете `UPDATE` в таблицу `ENROLLMENTS`, обновляя поле `progress_percent` на вычисленное значение. Теперь при запросе курса Frontend покажет актуальный прогресс!

### 3. Интеграция Smart Telegram-Бота (Пуш-Уведомления)
Логика общения с Telegram-ботом работает внутри нашего Go-монолита (без тяжелых брокеров сообщений).
1. **Привязка:** В Личном кабинете (фронтенд) студент нажимает "Привязать Telegram". Ссылка перекидывает его в бота с уникальным JWT или ID: `t.me/NeoflexEduBot?start=123useruuid`.
2. **Webhook:** Ваш Go-бэкенд получает от серверов Telegram JSON (Webhook) с командой `/start 123useruuid`.
3. **Сохранение:** Go парсит `123useruuid` из текста и делает привязку в БД: `UPDATE users SET tg_chat_id = X WHERE id = 123useruuid`.
4. **Уведомления:** В Алгоритме №1 (когда код проверен Judge0) и при принятии SVN Pull Request'а, бэкенд асинхронно через горутину вызывает API Телеграма `sendMessage` и поздравляет студента, повышая вовлеченность (gamification).

## Timeline реализации (Пошаговый план Backend-разработчика)

### Шаг 1: Поднятие сервера и Система Авторизации (Часы 1-4)
*Закладываем фундамент безопасности и Личный кабинет.*
- [ ] **1.1:** Установка `go get -u github.com/gin-gonic/gin` и создание `main.go`. Настроить CORS-middleware. Подключить БД пакетом `pgx` к Supabase.
- [ ] **1.2 Подключение Auth Middleware:** Написать функцию-хендлер для Gin `AuthMiddleware()`. В ней парсится заголовок HTTP и проверяется JWT подпись. При успехе: `c.Set("userID", id)` и `c.Set("role", role)`.
- [ ] **1.3 Логин:** Написать эндпоинт `POST /api/auth/login`. Найти юзера, сравнить пароли, сгенерировать JWT (библиотека `golang-jwt`) и вернуть фронту.
- [ ] **1.4 Профиль:** Написать защищенный эндпоинт `GET /api/users/me/profile`. Достает `userID` из сессии Middleware. Вытаскивает из `USERS` профиль, джоинит таблицу `USER_COMPETENCIES` и отдает фронтенду всю картину компетенций.

### Шаг 2: Эндпоинты агрегации (Read-Only API) (Часы 4-8)
*Эндпоинты потребления контента.*
- [ ] **2.1 (Для Студента):** Написать эндпоинт `GET /api/courses` (`SELECT * FROM courses WHERE is_published=true`).
- [ ] **2.2 (Для Модератора):** Написать `POST /api/courses/create`. Защитить проверкой: `if role != "moderator" { abort 403 }`. Это кабинет кураторов для сборки учебных модулей.
- [ ] **2.3:** Написать `GET /api/courses/:id/lessons` (выдает отсортированное дерево уроков для Плеера).

### Шаг 3: Детальная разработка Checker логики (Часы 8-14)
*Главная Wow-фича взаимодействия с внешним исполнителем.*
- [ ] **3.1:** Написать роут `POST /api/tasks/:task_id/check` (Обернуть в AuthMiddleware).
- [ ] **3.2:** Достать `reference_answer` из базы для `task_id` (404 Not Found, если таски нет).
- [ ] **3.3:** Сформировать HTTP-запрос к Judge0 с кодом пользователя в Payload.
- [ ] **3.4:** Распарсить ответ от Judge0. Ошибку типа `stderr` (в коде баг) отдать фронтенду как текст красной ошибки. Если успех, сравнить вывод с эталоном. 
- [ ] **3.5 Record:** Если всё ок, выполнить Insert статуса `success` в `SUBMISSIONS` и запустить инкрументацию баллов в `USER_COMPETENCIES` (Алгоритм №3). Ответить фронтенду обновленным процентом прогресса.

### Шаг 4: Развитие Админки и SVN-Листенер (Финальные часы)
*Интеграция с корпоративным репозиторием.*
- [ ] **4.1 SVN Listener:** Написать публичный маршрут `POST /webhooks/git`. Гитлаб или БитБакет будет стучаться сюда при мержах пулл-реквестов студентов.
- [ ] **4.2 Парсинг:** Добавьте структуру `struct` для маппинга ответа GitLab'a (важен тип экшена `action: "merged"` и заголовок PR `title: "Fix DEV-85"`).
- [ ] **4.3 Regex магия:** Напишите регулярное выражение `regexp.MustCompile(`DEV-\d+`)`, чтобы "поймать" ID задачи (например `DEV-85`) из названия репозитория/пулл-реквеста. По этому ID найдите студента, и выставьте ему `success` в базе.
- [ ] **4.4 Статистика для HR:** Если осталось время на хакатоне, напишите эндпоинт `GET /api/admin/users/stats` для Модератора, отдающий таблицу всех студентов, отсортированную по уровню прогресса `progress_percent`. Фронтенд нарисует красивую HR-панель.

### Шаг 5: Разработка Telegram-Бота (Опционально / Последние часы)
*Киллер-фича для презентации, которая добавляет проекту "живости".*
- [ ] **5.1 Регистрация:** Создайте бота в @BotFather в Telegram, получите API токен `TG_BOT_TOKEN`.
- [ ] **5.2 Webhook Engine:** Вместо Long-Polling настройте вебхук. Добавьте в Gin публичный роут `POST /api/tg/webhook`. Затем вызовите `setWebhook` метод API Telegram, чтобы он бомбил этот роут новыми сообщениями.
- [ ] **5.3 Обработка Deep-Link:** В роуте вебхука распарсите команду `/start`. Если в тексте передан UUID (Например: `/start f47a...`), выполните в базе `UPDATE users SET tg_chat_id = {chat_id_из_json}`.
- [ ] **5.4 Функция-Push:** Напишите простую функцию `func SendTgPush(userId string, tgChatId int, text string)` поверх обычного `net/http` клиента. Вызовите её в конце алгоритма Чекер логики и SVN листенера, чтобы присылать студенту пуш "Задание успешно проверено! Твой прогресс вырос на 10 баллов!".
