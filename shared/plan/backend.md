# Backend Спецификация (Golang)

Этот документ написан специально для Backend-разработчика с фокусом на кристальную простоту реализации в сжатые сроки хакатона. 

Архитектура построена по принципу **Thin Client**: фронтенд ничего не вычисляет. Ваша задача на Go — обращаться к базе данных, агрегировать данные, упаковывать всё в удобный JSON и отдавать фронтенду.

> **Важно:** 
> Настройка таблиц в Supabase и сложные тесты с анализом данных **не входят** в ваши задачи. Нужен максимально простой, железобетонно работающий алгоритм `if/else`, который не упадет во время презентации.

## Стек и Рекомендуемые Библиотеки (Go)
Инициализация проекта: `go mod init neoflex-lms`. 
- **Роутинг:** `github.com/gin-gonic/gin` — прост в освоении, идеально подходит для создания REST API.
- **Работа с PostgreSQL:** Пакет конфигурации `github.com/jackc/pgx/v5` или базовая ORM `gorm.io/gorm`.
- **Проверка кода (Система Judge0):** Боль и страдания написания собственных песочниц мы делегируем стороннему открытому API — **Judge0**. Это готовая платформа, которая поддерживает >50 языков программирования (в том числе SQL, Python). Она сама изолирует код и защищает от уязвимостей.
- **Data exchange:** Обычный встроенный `encoding/json`.

## Подробные алгоритмы работы (Бизнес-логика)

### 1. Как работает Code Checker (Интеграция с Judge0)
Писать свой локальный компилятор вы не будете, используем Judge0 как API.
**Алгоритм:**
1. Фронтенд присылает JSON: `{"user_id": "123", "task_id": "45", "user_code": "print('hello')", "language_id": 71}` *(71 - айди Python, 82 - SQL согласно документации Judge0)*.
2. Go-сервер берет `user_code` и через стандартный HTTP-клиент (пакет `net/http`) отправляет `POST` запрос во внешний API Judge0 на эндпоинт `/submissions?base64_encoded=false&wait=true`.
3. Judge0 безопасно выполняет этот код у себя и отвечает вам JSON-объектом, содержащим поле `stdout` (чистый текстовый вывод консоли от кода пользователя) или `stderr` (в случае ошибки).
4. Go делает SELECT из вашей БД Supabase: берет поле `reference_answer` для задачи с ID `45`.
5. Банальное сравнение: `if strings.TrimSpace(stdout) == strings.TrimSpace(reference_answer)`. 
6. Если сошлось — отдаете фронтенду: `{"status": "success", "console": stdout}`.
*(Примечание/Фолбэк: Если API Judge0 упадет в день презентации, в качестве запасного плана для SQL можно интегрировать пакет `github.com/xwb1989/sqlparser`, парсить строку на безопасность (запрет DROP) и дергать `db.Query()` у себя. Но Judge0 — приоритет №1).*

### 2. Как работает Аналитика (Прогресс и Матрица Компетенций)
**Подробный ход работы бекенда при начислении баллов:**
Никакой сложной высшей математики! Аналитика сводится к счетчикам очков. Когда Чекер в Алгоритме №1 сгенерировал `success` для студента, Go-бэкенд продолжает работу:
1. Вы запрашиваете у БД: к какому навыку или компетенции привязана эта задача? (`SELECT competency_id FROM tasks WHERE id = 45`).
2. Проверяете наличие записи у юзера: `SELECT level FROM user_competencies WHERE user_id = 123 AND competency_id = X`.
3. **Логика начисления:** Если записи еще нет — делаем `INSERT` с начальным `level = 10` (баллов). Если запись уже есть — делаем `UPDATE level = level + 10`. *Ограничение: `if level > 100 { level = 100 }`*.
4. **Общий прогресс курса:** Считаете метрику завершенности. Запрашиваете кол-во _всех_ задач в курсе и кол-во `success` задач у данного юзера в таблице `SUBMISSIONS`. Формула: `(Решено / Всего) * 100%`.
5. Делаете `UPDATE` в таблицу `ENROLLMENTS`, обновляя поле `progress_percent` на вычисленное значение. Теперь при запросе курса Frontend покажет актуальный прогресс!

## Timeline реализации (Пошаговый план Backend-разработчика)

### Шаг 1: Поднятие сервера и API Gateway (Часы 1-3)
*Задача: вывести приложение в сеть и вернуть первый JSON фронту.*
- [ ] **1.1:** Установка фреймворка `go get -u github.com/gin-gonic/gin` и создание `main.go`. Инициализация `r := gin.Default()`.
- [ ] **1.2:** Настроить CORS-middleware в Gin (без этого react-приложение упрется в браузерную безопасность и не сможет достучаться).
- [ ] **1.3:** Написать ручку `GET /ping`, возвращающую `{"status": "ok"}`. Убедиться, что фронтендер получает этот JSON.
- [ ] **1.4:** Подключение БД: Инициализировать подключение пакетом `pgx` к вашему облаку Supabase (DSN-строка лежит в переменных окружения `.env`).

### Шаг 2: Эндпоинты агрегации (Read-Only API) (Часы 3-6)
*Задача: Отдать Фронту массивы данных для витрины курса.*
- [ ] **2.1:** Написать эндпоинт `GET /api/courses` (`SELECT * FROM courses WHERE is_published=true`).
- [ ] **2.2:** Написать `GET /api/courses/:id/lessons`. Делает SELECT всех уроков (с сортировкой `ORDER BY order_index`), формирует массив слайсов в Go и отдает JSON.
- [ ] **2.3:** Написать `GET /api/users/me/competencies`. Извлекает `user_id`, читает таблицу `USER_COMPETENCIES` с джоином названий, отдает JSON формата `[{ "name": "SQL", "level": 25 }]`. 

### Шаг 3: Детальная разработка Checker логики (Часы 6-12)
*Главная логика взаимодействия с Judge0.*
- [ ] **3.1:** Написать роут `POST /api/check`. Создать структуру (struct) для парсинга входящего тела JSON `{task_id, code, language_id}`.
- [ ] **3.2:** Достать `reference_answer` из базы для конкретной Task с проверкой на 404 (задача не найдена).
- [ ] **3.3:** Написать функцию вызова внешнего API: сформировать POST HTTP-запрос к Judge0 с кодом пользователя в теле запроса (payload).
- [ ] **3.4:** Распарсить ответ от Judge0. Если есть поле ошибки компиляции `stderr` — вернуть статус `failed` фронтенду, показав ему ошибку. Если всё окей, сравнить `stdout` с `reference_answer`. 
- [ ] **3.5:** Сформировать финальный ответ фронту, вставить запись (Insert) попытки студента в таблицу БД `SUBMISSIONS`. Запустить пересчет прогресса (Пункт "Аналитика" из Бизнес-логики вгорутине или синхронно).

### Шаг 4: Sandboxes (Песочницы для Студентов) (Часы 12-16)
*Вторая Wow-фича. Зачем нужен Docker: для корпоративного обучения часто нужно дать ученику "живой" линукс-сервер, чтобы он выполнил команды руками (настроил базу, накатил зависимости). Выдавать всем студентам одну виртуалку нельзя — сломают. **Docker** за 2 секунды поднимает изолированный мини-сервер из чистого шаблона каждому студенту.*
- [ ] **4.1 Обертка CLI:** Если вы глубоко не знаете Docker SDK для Go — не страдайте! Используйте встроенный пакет ОС `os/exec` для вызова обычных bash-команд в консоли прямо из Go (команда `exec.Command`).
- [ ] **4.2 Создание:** Написать роут `POST /api/sandbox/start`. Выполнить из Go: `docker run -d -P ubuntu_sandbox_image`. Эта команда запустит контейнер в фоне и вернет `container_id`.
- [ ] **4.3 Парсинг порта:** Выполнить из Go: `docker port <container_id>`. Сохранить полученный сгенерированный порт (шлюз) на который отобразился контейнер.
- [ ] **4.4 Выходные данные:** Отдать фронту JSON: `{"url": "http://IP-ВАШЕГО-БЭКА:порт"}`. И сохранить эту привязку в БД `SANDBOXES` с полем время жизни `expires_at`.
- [ ] **4.5 Уничтожение (GC):** Запустить отложенную горутину `go func() { time.Sleep(30 * time.Minute); exec.Command("docker", "rm", "-f", container_id) }()` для автоудаления.

### Шаг 5: SVN/Git Листенер (Финальные часы)
*Интеграция с корпоративным репозиторием (Принятие Pull Requests от студентов).*
- [ ] **5.1:** Предложите команде выбрать корпоративный сервер (Gitea/Gitlab/GitHub). 
- [ ] **5.2:** Написать HTTP роут `POST /webhooks/git` в вашем сервере API. Этот роут вы укажете в настройках репозитория. Ваш сервер будет ловить каждый "Push" или "Pull Request" студентов.
- [ ] **5.3 Парсинг Webhook'а:** Докер/Гитлаб пришлет вам JSON. Добавьте в Go `struct` поля для парсинга события (например `action: "merged"` и заголовок PR `title: "DEV-85 fixes"`).
- [ ] **5.4 Магия Regex:** Напишите регулярное выражение `regexp.MustCompile(`DEV-\d+`)`, чтобы "поймать" ID задачи (например `DEV-85`) из названия мерж-реквеста.
- [ ] **5.5 Завершение:** Если этот Pull Request был принят (merged == true), сделайте UPDATE статус этой таски в таблице `SUBMISSIONS` на "success". И автоматом вызовите уже написанную функцию начисления баллов компетенций из Шага 2!
