# Стратегия Тестирования и План Защиты (QA Plan)

Так как разработка в рамках хакатона сильно ограничена по времени, мы используем **простые, но максимально эффективные методы**. Главная цель — доказать жюри надежность архитектуры, безопасность сервиса автопроверки и работоспособность Ролевой Модели. 

Мы отказываемся от дорогих по времени UI-тестов (Cypress/Selenium). Выберем следующий пирамидальный подход:

## 1. Автоматизированное API Тестирование (TypeScript + Jest + Axios)
Самый эффективный способ убедиться, что сервер работает корректно — написать внешние E2E (End-to-End) скрипты опроса API. Так как весь проект строится на современном стеке веб-технологий, QA-инженер (или разработчик) создаст небольшую отдельную папку `qa/` и проинициализирует там TypeScript-проект.

**Стек для тестов:** `npm i -D jest ts-jest @types/jest axios`

Внутри мы пишем простые `api.test.ts` файлы, которые делают HTTP-вызовы на локальный (или задеплоенный) бэкенд:

* **Тест Авторизации (RBAC Guard):**
  * Скрипт отправляет `axios.get('/api/users/me/profile')` без токена. Ожидаем ответ со статусом `401 Unauthorized`.
  * Скрипт отправляет `axios.post('/api/courses/create')` (создание курса) с токеном *студента*. Ожидаем `403 Forbidden`. Это докажет жюри зрелость архитектуры безопасности.
* **Тест парсера SVN-коммитов (Regex):**
  * Эмулируем вызов `axios.post('/webhooks/git')` с различными заголовками пулл-реквестов: `["Fix DEV-85", "just a merge"]`. Убеждаемся, что алгоритм возвращает статус 200 только для правильных паттернов, игнорируя системные коммиты.
* **Тест Калькулятора Прогресса:**
  * Вызываем таску через Чекер, которая должна дать +10 очков компетенции. Проверяем, что если у пользователя уже 95 очков, то API после инкремента в профиле вернет хард-лимит `100`, чтобы студент не мог "перекачаться".

## 2. Интеграционные тесты Уязвимостей (Система Judge0)
Самый критичный блок системы — тот, где выполняется код неопытных или злонамеренных студентов. Эти сценарии мы будем показывать на защите!
* **Тест-кейс "Бесконечный Цикл (Time Limit Exceeded)":**
  * Отправляем Python код: `while True: pass`.
  * *Ожидаемый результат:* API Judge0 прерывается по установленному таймауту выполнения (например 2 секунды), наш Go-сервер аккуратно перехватывает ошибку, не зависает навсегда и отдает фронтенду статус `failed` с ошибкой "Time Limit".
* **Тест-кейс "SQL Уязвимость (DROP)":**
  * Отправляем код: `DROP TABLE users; UPDATE auth SET role='admin';`.
  * *Ожидаемый результат:* Во-первых, Judge0 запустит этот запрос в полностью пустой сторонней контейнерной базе. Она никак не связана с нашей Supabase БД! Наша реальная база в полной безопасности. Во-вторых, ответ (stdout) будет пуст или будет содержать ошибку. Он не совпадет с `reference_answer` от нашего Бэкенда, и система выставит статус `failed`.

## 3. Сценарий Ручного Демо для Жюри (End-to-End)
Это четкий QA-скрипт или сценарий (сцена) для финального питча. Он проходится вручную и показывает весь Data Flow:

1. **Регистрация:** Вход в систему под чистым Студентом.
2. **Пустой Радар:** Заход в Личный кабинет. Мы показываем жюри Матрицу компетенции — там по нулям (нет баллов).
3. **Telegram:** Студент "привязывает" Telegram-аккаунт через Deep-Link ссылку. (На бэкенд улетает вебхук и вяжет `tg_chat_id`).
4. **Практика:** Переход в плеер лекций. На экране Monaco Editor (IDE). Студент вводит верный SQL-запрос и нажимает зеленую кнопку "Run". 
5. **WOW Эффект №1:** Сервер мгновенно отвечает `success` и на фронтенде стреляет анимация "Конфетти" (Confetti library).
6. **WOW Эффект №2:** На телефон студенту приходит пуш-уведомление от Телеграм-Бота платформы: *"Поздравляем! Задача решена верно!"*.
7. **Финал:** Перезагружается Радар в Личном кабинете, наглядно демонстрируется прирост навыка (всплеск +10 баллов по оси SQL).
