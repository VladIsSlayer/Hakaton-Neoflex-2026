# Стратегия Тестирования и План Защиты (Testing Plan)

Так как разработка в рамках хакатона сильно ограничена по времени, QA и тестирование фокусируется в первую очередь на демонстрации жюри работоспособности концепции "Изолированного ИТ-Стенда" (Proof of Concept, PoC). Главная задача: убедиться, что система отказоустойчива к вредоносным/ошибочным действиям студента.

Ожидается, что на проекте не будет покрытия в 100% юнит-тестов (за нехваткой времени), но ключевые узлы должны быть доказаны тестами (или скриптами проверки).

## 1. Unit Тесты и Изоляция Логики (Backend)
Критическая уязвимость микросервисной системы: "Code Checker", где выполняется чужой код. Именно здесь необходимы пруф-тесты:
- **Тест-кейс "SQL Инъекция / Зловред":** 
  - Направляем на проверку `DROP TABLE users;` или `DELETE FROM COURSES;`. 
  - Ожидаемый результат: Запрос исполняется под read-only (созданным только на select) пользователем базы, попытка падает, таблица `Submissions` получает статус `failed`, реальные данные проекта остаются невредимыми.
- **Тест-кейс "Time Limit Exceeded (Исполнение бесконечного кода)":**
  - Пользователь отправляет `while True: pass` на Python.
  - Ожидаемый результат: Микросервис-экзекьютор убивает процесс/контейнер через таймаут (2-5 сек), и отдает юзеру `TLE Error` без зависания всего Го-сервера Gateway.

## 2. API Тестирование и Безопасность Supabase (Data Layer)
- **RLS Security Check (Контроль доступа):**
  - Присылаем `HTTP /GET` для получения `SUBMISSIONS`, авторизовавшись JWT-токеном пользователя (Student 1).
  - Студент 1 должен получить только 1 запись своих отправок и не увидеть записи Студента 2.
  - Пытаемся сделать `UPDATE` курса без токена Админа/HR. Ожидается `403 Forbidden`.

## 3. End-to-End Демо (Ключевые сценарии для жюри)
Прохождение пути студента (Сценарий, который должен работать без багов во время презентации):
1. Вйти в систему (логин).
2. Перейти в "Каталог курсов" -> Вступить на курс "Основы SQL/Основы K8s".
3. Выбрать практический урок (Где справа расположен Monaco Editor). Стереть дефолтный код на неправильный, нажать RUN. Дождаться красной панели ошибки в выводе консоли. Зафиксировать `failed` в БД.
4. Исправить на верный ответ, идентичный эталону (reference). Нажать RUN. Система выдает статус SUCCESS.
5. Демонстрация работы **Матрицы компетенции** — сразу после пункта 4, перейдя в Dashboard/Profile юзер видит, что индикатор (допустим SQL скилл пополнился на +10 очков компетенции). 
6. Демонстрация **Sandobox (Изолированные стенды)**: Запустить Урок 2 "Администрирование Nginx", получить URL от `Sandbox Manager`, открыть в браузере стартовую страничку Nginx и показать, что это живой настоящий контейнер.

## 4. Требования к CI/Deploy (Для тестов)
- Наличие `docker-compose.yml`, который поднимает все 5 микросервисов + Frontend одной командой `docker-compose up -d --build`. 
- Это позволяет жюри или тестерам скачать код с гитхаб корпоративного репозитория NeoFlex, развернуть на машине полностью рабочий ландшафт и прокликнуть e2e путь.
