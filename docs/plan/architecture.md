# Общая Архитектура Проекта NEO EDU

## Описание системы
Проект представляет собой специализированную систему управления обучением (LMS), адаптированную под потребности ИТ-отрасли. Ключевая концепция — модульность и возможность интеграции специфичных ИТ-инструментов (таких как авто-чекеры кода) непосредственно в учебный процесс. Архитектура построена на микросервисном подходе, что позволяет легко масштабировать отдельные узлы системы, заменять компоненты и строить отказоустойчивый контур.

## Технологический стек
*   **Твёрдый фундамент данных (BaaS):** Supabase (содержащий в себе PostgreSQL, системы Auth и Storage).
*   **Backend Микросервисы:** Go (опционально, если нужно — FastAPI) для всех асинхронных тяжелых операций и роутинг через REST / gRPC.
*   **Изолированная проверка кода:** Использование внешнего API Judge0.
*   **Frontend порталы:** React SPA (Vite или Next.js).

## Задействованные компоненты системы

### 1. Базовый уровень хранения: Supabase
Supabase выступает центральным узлом для хранения состояния всей системы, статики и управления доступом:
*   **PostgreSQL:** Хранение данных пользователей, каталога курсов, уроков и попыток прохождений задач (`Submissions`).
*   **Auth (JWT & RLS):** Управление авторизацией и ролевой моделью. Позволяет frontend-приложениям безопасно и напрямую общаться с БД с соблюдением политик безопасности данных (Row Level Security - RLS).
*   **PostgREST API:** Автоматически генерируемый API для базовых CRUD-операций с курсами и пользователями. Значительно разгружает Backend.

### 2. Пользовательские интерфейсы (Frontend на React)
Разделен на логические панели администрирования:
*   **Портал Студента:** Плеер для прохождения видео-материалов, отслеживания личного прогресса матрицы ИТ-компетенций и работы во встроенной браузерной IDE для выполнения практических зданий (SQL, алгоритмы).
*   **Панель Администратора / HR:** Конструктор модулей, просмотр сводки по компетенциям, управление доступом контент-мейкеров. Учитывает гайдлайны и строгий IT-стиль компании NeoFlex.

### 3. Backend Микросервисы (Сложная бизнес-логика)
Специфичная логика, которую небезопасно или невозможно отдать напрямую в Supabase, оркестрируется через API Gateway:
*   **Code/SQL Checker:** Запуск кода пользователя для верификации и проверки правильности исполнения (сравнивается с `reference_answer`).
*   **Auth & Role Manager:** Компонент (встроенный в Gateway), занимающийся генерацией, распознаванием JWT-токенов и распределением прав между Студентами и Модераторами.
*   **Git/SVN Reviewer:** Микросервис-листенер вебхуков, получающий события (events) о создании Pull Requests в корпоративной системе репозиториев.
*   **Analytics & Matrix:** Сервис, следящий за прогрессом («отставание» или «опережение» графика) и пересчитывающий матрицу ИТ-компетенций учащегося на основе выполненных Задач.
*   **Smart Telegram Bot:** Асинхронный бот-помощник для трекинга успеваемости, интегрированный в сам профиль LMS.

## Поток данных (Data Flow)
1. **Чтение справочной информации:** Студент открывает портал, Frontend забирает данные каталога напрямую из **Supabase** (PostgREST API) используя свой токен аутентификации.
2. **Отправка решения:** На уроке "Код", Frontend отправляет скрипт (payload) на **API Gateway**. Gateway маршрутизирует вызов на **Code/SQL Checker**. 
3. **Обработка и Запись:** Checker безопасно прогоняет скрипт, пишет статус "success" вместе с логами работы в таблицу БД `Submissions` через Service Account. Фронтенд слушает изменения БД и показывает юзеру зеленую панель успеха.
4. **Реактивность:** Микросервис **Analytics Service**, отловив появление нового решения, пересчитывает скиллы студента и повышает ему `level` в таблице `USER_COMPETENCIES`.
