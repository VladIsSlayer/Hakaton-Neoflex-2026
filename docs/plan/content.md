# План Наполнения Контентом (Mock Data & Seeding)

Чтобы проект на финальной защите (Демо) выглядел "живым" и готовым к коммерческому использованию, нам необходимо заранее подготовить контентную базу. Пустые экраны и графики могут испортить впечатление от крутой архитектуры.

Эта работа поручается отдельному участнику команды (например, системному аналитику или менеджеру) и выполняется параллельно с написанием кода. Делать это можно прямо через красивый интерфейс **Supabase Table Editor**, прокликивая таблицы как в Excel, без написания скриптов.

## 1. Справочные данные (Заполняем первыми)

### Компетенции (`COMPETENCIES`)
Заполняем ИТ-скиллами, горячо востребованными в компании NeoFlex:
1. `SQL & Инженерия Данных`
2. `Алгоритмика (Python)`
3. `DevOps & Git`
4. `Архитектура Приложений`

Создаем как минимум 2 логических урока с контентом:

* **content_body:** (Разметка Markdown) *Короткий текст о том, что такое сортировка пузырьком и встроенные функции Python.*
* **order_index:** `1`

**Урок 2:**
* **title:** `Шаг 2. Выборка данных (SQL)`

## 3. Интерактивные Задачи и Референсы (`TASKS`)
Это самая важная для нас таблица. От неё будет отталкиваться **Judge0 Checker**.

**Задача к Уроку 1 (Алгоритмы):**
* **Текст на фронтенде (в UI):** "Напишите скрипт, который сортирует массив `[5, 2, 8, 1]` и выводит результат в консоль."
* **language_id:** `71` (Код языка Python в системе Judge0).
* **reference_answer:** `[1, 2, 5, 8]`
* *Логика для демо:* Если студент напишет `print(sorted([5, 2, 8, 1]))`, вывод точно совпадет с референсом. Бэкенд начислит баллы к скиллу: `Алгоритмика (Python)`.

**Задача к Уроку 2 (SQL):**
* **Текст на фронтенде:** "Выведите все данные из таблицы."
* **language_id:** `82` (Код языка PostgreSQL для Judge0).
* **reference_answer:** *(Тут вписывается точный консольный вывод, который ожидает Judge0 при селекте)*
* *Логика для демо:* Начислит баллы к скиллу: `SQL & Инженерия Данных`.

## 4. Фейковые Пользователи для HR-Панели

Чтобы во время презентации Модератор (HR) имел красивые графики со статистикой (вместо одного-единственного юзера), в базу заранее вносим "виртуальных" студентов.

В таблицы `USERS`, `USER_COMPETENCIES` и `ENROLLMENTS` вписываем:
1. **Иван Петров (Лидер):** `progress_percent = 90%` (Высокий уровень, прокачан Python на 80 баллов).
2. **Анна Сергеева (Середнячок):** `progress_percent = 45%` (Средний уровень, высокий скилл SQL, но провал по Алгоритмам).
3. **Григорий Воронов (Отстающий):** `progress_percent = 10%`.

**Wow-эффект на защите:** Когда жюри смотрит на админку под аккаунтом Модератора, мы показываем, как легко сортировать сотрудников по компетенциям и выявлять слабое звено в обучении, основываясь на готовых данных.
